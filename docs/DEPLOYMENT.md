# Deployment

This document covers how the Tolka developer documentation site is built, deployed, and maintained.

## Architecture

```
GitHub (tolka-health/docs)
  │
  ├─ PR → CI workflow (.github/workflows/ci.yml)
  │       Build, type check, OpenAPI lint, link check, PHI lint, API key scan
  │
  └─ Push to main → Deploy workflow (.github/workflows/deploy.yml)
          Build → Cloudflare Pages (docs.tolka.health)
```

- **Hosting:** Cloudflare Pages (static, edge-cached)
- **Domain:** `docs.tolka.health`
- **Framework:** Astro 5 + Starlight 0.32 (static output)
- **Build output:** `./dist` (pure HTML/CSS/JS, no server runtime)

## Environments

| Environment | Trigger | URL |
|-------------|---------|-----|
| Production | Push to `main` | `docs.tolka.health` |
| Preview | Pull request | Auto-generated by Cloudflare Pages per PR |
| Local dev | `pnpm dev` | `http://localhost:4321` |

## Required secrets (GitHub repo settings)

| Secret | Purpose |
|--------|---------|
| `CLOUDFLARE_API_TOKEN` | Cloudflare Pages deploy token |
| `CLOUDFLARE_ACCOUNT_ID` | Cloudflare account identifier |

These are configured in **Settings > Secrets and variables > Actions** on the GitHub repo.

## Required variables (GitHub repo settings)

| Variable | Purpose | Values |
|----------|---------|--------|
| `PUBLIC_REPO` | Enables "Edit on GitHub" links | `false` (default) or `true` when repo goes public |

Configured in **Settings > Secrets and variables > Actions > Variables**.

## Local development

**Requirements:** Node.js >= 20, pnpm >= 9

```bash
pnpm install          # install deps + activate pre-commit hook
pnpm dev              # dev server at http://localhost:4321
pnpm build            # production build → ./dist
pnpm preview          # serve production build locally
```

## CI pipeline (pull requests)

Runs on every PR targeting `main`. All checks must pass before merge.

**Job: Build & validate**

| Step | Command | What it checks |
|------|---------|---------------|
| Type check | `pnpm check` | Astro/TypeScript errors |
| Build | `pnpm build` | MDX parsing, broken imports, Starlight config |
| OpenAPI lint | `pnpm dlx @redocly/cli lint public/openapi/tolka-v1.yaml` | Spec validity |
| Link check | `pnpm dlx astro-link-checker --base-dir ./dist --ignore-external` | Broken internal links |

**Job: PHI content lint**

| Step | What it checks |
|------|---------------|
| PHI patterns | Rejects `patient reports`, `history of`, `chief complaint`, `pasient rapporterer`, date patterns (`DD.MM.YYYY`) in MDX files |
| API key scan | Rejects `sk_live_*` patterns in `src/` and `public/` |

## Deploy pipeline (push to main)

Runs automatically when a PR is merged to `main`.

1. Install dependencies (`pnpm install --frozen-lockfile`)
2. Build (`pnpm build` with `PUBLIC_REPO` env var)
3. Deploy `./dist` to Cloudflare Pages via `cloudflare/pages-action@v1`

Typical deploy time: ~1 minute from merge to live.

## OpenAPI spec sync

The API spec at `public/openapi/tolka-v1.yaml` is synced from the core API repo via `.github/workflows/openapi-sync.yml`.

**Trigger:** `repository_dispatch` event (`openapi-spec-update`) from the core API repo on tagged releases.

**Validation before sync:**
1. PHI content lint (rejects patient data, long examples, fødselsnummer patterns)
2. Redocly OpenAPI validation
3. Sandbox-only URL check (blocks non-sandbox production URLs)

**Result:** Opens a PR with the updated spec for human review before merge.

**Never edit `public/openapi/tolka-v1.yaml` manually.** Changes flow from the core API repo only.

## Pre-commit hook

A local pre-commit hook at `.hooks/pre-commit` runs automatically (activated by `pnpm install` via the `prepare` script). It scans staged files for:

- PHI patterns in content files
- PII (fødselsnummer, date formats)
- Production API keys (`sk_live_*`)
- Business IP (scoring function names, prompt internals, calibration thresholds)
- Production URLs in the OpenAPI spec
- Local filesystem paths

## Branch protection

`main` is protected:
- Requires passing CI
- Requires at least one maintainer review
- Cloudflare Pages creates a preview deployment for every PR

## Rollback

Cloudflare Pages keeps previous deployments. To roll back:

1. Go to **Cloudflare Dashboard > Pages > tolka-health-docs > Deployments**
2. Find the last known-good deployment
3. Click **Rollback to this deployment**

This takes effect immediately at the edge. No git revert required for urgent rollbacks.

## Monitoring
