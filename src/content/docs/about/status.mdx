## Health endpoints

The API exposes two unauthenticated health endpoints suitable for monitoring tools:

| Endpoint | Purpose | 200 means |
|----------|---------|-----------|
| `GET https://api.tolka.health/health` | Liveness | The API process is running and serving traffic |
| `GET https://api.tolka.health/readyz` | Readiness | Liveness + Supabase metadata store is reachable |

Both endpoints return `{"status":"ok"}` (or `{"status":"ready","database":true}`) on success and HTTP `503` on failure. Probe `/health` aggressively (1–5 s) and `/readyz` more sparingly (5–10 s) — `/readyz` makes a real database call that is cached for 3 s.

```bash
curl -s https://api.tolka.health/health
# {"status":"ok"}

curl -s https://api.tolka.health/readyz
# {"status":"ready","database":true}
```

## What to do during an outage

1. Confirm the symptom with `curl https://api.tolka.health/health` — if it returns 5xx, the API is down.
2. Implement your fallback UI (see [Error Handling](/sdks/web/error-handling/)) — never show an empty translation to a patient.
3. Use exponential backoff with jitter for retries — do not hammer the API during recovery.
4. Reach out to [Support](/about/support/) for incident updates.
