`POST /v1/medical/session` creates an encrypted session token that carries context across multiple translation calls in a single clinical encounter.

For the full schema, see the [Interactive Reference](/api-reference/index/).

---

## When to create a session

Create a session when:
- You are starting a new clinical consultation that will involve multiple translated exchanges
- You want to simplify repeated calls (no need to pass `domain`, `source_lang`, `target_lang` each time)
- You need conversation continuity (the model is aware of previous exchanges in the session)

---

## Session lifecycle

```
POST /v1/medical/session → stk_enc_... (valid for ttl_minutes)
        ↓
POST /v1/medical/translate + session_token → translation
POST /v1/medical/translate + session_token → translation
        ↓
Token expires after ttl_minutes (default: 60)
        ↓
Create new session for next encounter
```

---

## Request fields

| Field | Type | Required | Default | Description |
|-------|------|----------|---------|-------------|
| `domain` | string | Yes | — | Clinical domain for all translations in this session |
| `source_lang` | string | Yes | — | BCP 47 source language |
| `target_lang` | string | Yes | — | BCP 47 target language |
| `ttl_minutes` | integer | No | 60 | Session duration: 15–480 |

## Response fields

| Field | Type | Description |
|-------|------|-------------|
| `token` | string | Encrypted session token (prefix: `stk_enc_`) |
| `expires_at` | string | ISO 8601 timestamp when the token expires |
| `session_id` | string | Opaque identifier for logging/support |

---

## Security model

Session tokens are encrypted opaque blobs. The encryption key lives server-side. A token can only be used with the same API key that created it — it cannot be transferred to another key or organisation.

Store tokens in memory only — not in cookies, `localStorage`, or `sessionStorage`. See [Sessions concept](/concepts/sessions/) for full security guidance.
